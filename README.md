# Datawhale 学习笔记：零基础在 AMD 云平台体验 Gemma 4 模型

> 标签：#AMDev #Datawhale #ROCm #AMD #魔搭社区 #ModelScope #Gemma #vLLM #大模型

## 1. 学习背景

这次 Datawhale 的任务不是本地安装大模型，也不是调用线上 API，而是让零基础同学进入 AMD 提供的云平台环境，借助魔搭社区 ModelScope 下载模型，并用 vLLM 把模型服务跑起来，最后在终端里和模型进行一次对话测试。

对我来说，这次练习最重要的收获是：第一次把“大模型文件”变成了一个真正可以对话的“模型服务”。以前理解大模型时，更多停留在“网页聊天”或“别人已经搭好的产品”层面；这次操作之后，我更清楚地知道了模型部署的大致链路。

整个流程可以概括为：

```text
进入 AMD 云平台
        ↓
打开云端 Notebook / Terminal
        ↓
通过 ModelScope 下载 Gemma 4 模型文件
        ↓
安装并启动 vLLM 推理服务
        ↓
新开终端连接服务并测试对话
```

## 2. 云平台体验流程

### 2.1 进入云端环境

任务提供的是已经配置好的云端环境，所以不用自己准备显卡、驱动、Linux 服务器，也不用在本地折腾复杂依赖。进入平台后，主要是在 JupyterLab 风格的页面中操作。

页面里可以看到文件区、Notebook、Terminal 等入口。对于小白来说，最关键的是先找到可以输入命令的 Terminal，因为后续下载模型、安装依赖、启动服务都要在终端里完成。

这一步让我意识到：大模型实践不一定一开始就要自己买显卡或搭服务器，云平台可以把底层硬件和环境先准备好，让学习者把注意力放在模型体验和部署流程上。

### 2.2 下载 Gemma 4 模型

任务中使用的是 Gemma 4 模型，并通过魔搭社区 ModelScope 进行下载。ModelScope 可以理解成国内比较常用的模型下载与托管平台，优势是访问速度相对稳定，更适合在国内网络环境中拉取模型文件。

下载成功后，截图中可以看到模型目录下出现了多种文件，例如：

- `config.json`
- `tokenizer.json`
- `tokenizer_config.json`
- `model.safetensors`

其中最核心的是模型权重文件，也就是保存模型参数的文件。模型本身并不是一个会“主动聊天”的软件，它更像是一大堆已经训练好的参数。只有把它加载到推理框架里，才能真正对外提供问答能力。

## 3. 启动 vLLM 推理服务

下载好模型后，下一步是用 vLLM 启动推理服务。vLLM 是一个大模型推理框架，可以把模型加载起来，并对外提供类似服务接口的能力。

任务中先更新了云环境里的相关依赖，例如：

```bash
uv pip uninstall torchvision
uv pip install vllm torchvision \
  --no-cache \
  --index-url https://mirrors.aliyun.com/pypi/simple/ \
  --extra-index-url https://wheels.vllm.ai/rocm/ \
  -U
```

这里我理解到两个点：

- 云平台虽然已经准备了大部分环境，但仍然可能需要根据模型和硬件更新依赖；
- AMD 环境下会涉及 ROCm 相关依赖，所以安装命令里使用了 vLLM 的 ROCm wheel 地址。

安装完成后，使用下面的命令启动模型服务：

```bash
vllm serve ./models/google/gemma-4-E4B-it/ --served-model-name gemma-4-E4B-it
```

运行后需要等待一段时间。截图中提示大约等几分钟，看到类似下面的标志就表示服务启动成功：

```text
Application startup complete.
```

这里有一个很重要的注意点：启动 vLLM 服务的这个终端会被模型服务占用，不能关闭，也不要按 `Ctrl+C`。如果关闭了这个终端，模型服务就会停止。

## 4. 新开终端测试对话

因为第一个终端已经在运行模型服务，所以需要新开一个 Terminal，用它来连接刚才启动的服务并发送测试问题。

测试命令如下：

```bash
vllm chat --url http://localhost:8000/v1 --model gemma-4-E4B-it
```

这里的含义是：

- `localhost:8000`：访问当前云环境里正在运行的本地服务；
- `/v1`：服务接口路径；
- `--model gemma-4-E4B-it`：指定刚才启动时命名的模型；
- `vllm chat`：用命令行方式和模型聊天。

如果连接失败，优先检查第一个终端里的 vLLM 服务是否还在运行，以及是否已经出现 `Application startup complete.`。如果服务还没启动完成，就需要继续等待。

## 5. 我对关键概念的理解

### 5.1 大模型是什么

这次任务里有一个很形象的解释：大模型的核心工作是根据前面的文字预测下一个最可能出现的词。它不是像传统程序那样被程序员写死规则，而是在大量文本中学习语言规律、知识关联和常见表达方式。

所以大模型能聊天、写作、总结、推理，本质上都是在利用训练得到的参数进行计算。

### 5.2 参数和权重

模型文件里保存的是大量参数，也可以叫权重。参数越多，模型通常越有可能表现出更强的能力，但也会需要更多显存和计算资源。

任务里提到的 E4B 可以理解为 40 亿参数这个量级。它比超大模型更适合入门体验，因为对硬件要求相对低一些，但又足够展示大模型的基本能力。

一句话理解：

```text
模型 = 一堆参数
推理 = 用这些参数进行计算
部署 = 把模型放到服务里，让别人可以访问
```

### 5.3 ModelScope 的作用

ModelScope 在这次任务中主要承担“下载模型”的角色。它相当于一个模型社区和模型仓库，让我们可以比较方便地获取开源模型文件。

如果没有这类平台，就需要自己到不同来源找模型、处理下载速度和文件完整性问题。对初学者来说，ModelScope 降低了很多门槛。

### 5.4 vLLM 的作用

模型文件本身只是静态文件，不能直接聊天。vLLM 的作用就是把模型加载起来，让它真正跑起来，并提供可以访问的推理服务。

可以把它理解成：

```text
模型文件是“食材”
vLLM 是“厨房和厨师”
终端里的聊天命令是“点菜”
模型输出就是“做好的菜”
```

这个比喻虽然简单，但帮助我理解了为什么要先启动服务，再新开终端测试。

## 6. 踩坑记录

### 6.1 第一个终端不能关闭

启动 vLLM 后，第一个终端会一直显示服务日志。刚开始我会误以为命令还没结束，其实这是正常现象。只要服务在运行，这个终端就会一直被占用。

正确做法是保持第一个终端运行，然后再打开第二个终端测试对话。

### 6.2 要等服务真正启动完成

模型加载需要时间，尤其是第一次启动时。不能看到命令开始执行就立刻去测试，否则可能出现连接失败。

比较可靠的判断方式是等待日志中出现：

```text
Application startup complete.
```

### 6.3 模型下载要确认文件完整

下载模型后，需要确认模型目录中已经出现完整文件。如果模型文件缺失，后面启动 vLLM 时就可能报错。

截图里通过查看目录文件的方式确认了 Gemma 4 模型文件已经下载成功，这是一个很好的检查动作。

### 6.4 依赖安装失败时不要盲目重试

任务中也提到，如果 ModelScope 没安装成功，可以重新执行：

```bash
pip install -U modelscope
```

我的理解是，遇到依赖问题时要先看报错信息，再判断是包没装、版本不对，还是网络源有问题。盲目重复执行命令可能会让环境更乱。

## 7. 这次实践让我真正理解了什么

这次练习最大的价值，是让我把几个以前模糊的词串起来了：

- 模型：训练后得到的一堆参数；
- 权重：参数的另一种说法；
- 推理：用训练好的模型生成结果；
- 部署：把模型放到服务中，让外部可以访问；
- ModelScope：下载模型的社区和渠道；
- vLLM：让模型高效跑起来的推理框架；
- 云平台：提供显卡、系统和基础环境的地方。

以前看到“大模型部署”会觉得很抽象，现在可以理解为一句话：

```text
把下载好的模型文件，用 vLLM 在云服务器上跑成一个可以访问的服务。
```

## 8. 总结

通过这次 Datawhale 任务，我完成了一次从云平台进入、模型下载、服务启动到终端对话测试的完整体验。虽然命令看起来不少，但主线其实很清楚：先准备模型，再启动推理服务，最后连接服务进行对话。

对零基础学习者来说，这个任务很适合作为大模型部署的第一课。它不要求一开始就理解所有底层细节，但能让人亲手看到模型是怎么从一个文件夹里的权重文件，变成一个可以交互的聊天服务。

后续如果继续学习，我想进一步尝试：

- 修改提示词，观察模型输出变化；
- 对比不同大小模型的速度和效果；
- 了解 vLLM 的并发和显存优化；
- 尝试用简单 Web 页面调用本地模型服务。
